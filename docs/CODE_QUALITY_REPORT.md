## 🔍 详细评估

### 1. 代码结构 (75/100)

#### ✅ 优点
- **模块化设计**: 后端采用清晰的 MVC 分层 (routers/services/db/utils)
- **职责分离**: 数据获取、业务逻辑、API 路由分离明确
- **配置集中**: `data_sources.py` 统一管理外部数据源
- **前端组件化**: Vue 组件按功能拆分 (DashboardGrid, NewsFeed, IndexLineChart 等)

#### ⚠️ 问题
| 严重程度 | 问题描述 | 位置 |
|----------|----------|------|
| 🔴 高 | 部分文件过长 (`data_fetcher.py` 1162+ 行) | `backend/app/services/data_fetcher.py` |
| 🟡 中 | `market.py` 职责过重 (900+ 行，含路由+业务逻辑) | `backend/app/routers/market.py` |
| 🟡 中 | 前端 `IndexLineChart.vue` 过于复杂 (670+ 行) | `frontend/src/components/IndexLineChart.vue` |
| 🟢 低 | 存在未使用的导入和代码 (如 `market_mock.py`) | `backend/app/routers/market_mock.py` |

#### 📋 改进建议
1. **拆分大文件**: 将 `data_fetcher.py` 按数据源拆分为多个模块
2. **提取业务逻辑**: 将 `market.py` 中的数据处理逻辑移至 service 层
3. **组件拆分**: 将图表逻辑从 `IndexLineChart.vue` 提取到 composables

### 2. 代码规范 (68/100)

#### ✅ 优点
- **命名规范**: 函数/变量命名基本符合 Python PEP8 规范
- **类型提示**: 部分函数使用了类型注解
- **文档字符串**: 关键函数有 docstring 说明
- **前端规范**: Vue 组件结构清晰，使用 Composition API

#### ⚠️ 问题
| 严重程度 | 问题描述 | 示例 |
|----------|----------|------|
| 🔴 高 | 混合使用单双引号，缩进不一致 | `data_fetcher.py` 多处 |
| 🔴 高 | 行尾空格、空行不规范 | 多个文件 |
| 🟡 中 | 部分函数缺少类型注解 | `market.py` 中大量函数 |
| 🟡 中 | 魔法数字硬编码 | `_MACRO_CACHE_TTL = 600` 等 |
| 🟢 低 | 注释风格不统一 | `# ──` vs `# ---` |

#### 📋 改进建议
1. **引入 lint 工具**: 配置 `black` + `isort` + `flake8` 自动格式化
2. **类型检查**: 逐步添加 `mypy` 类型检查
3. **常量提取**: 将魔法数字提取到配置文件中

---

### 3. 异常处理 (65/100)

#### ✅ 优点
- **全局异常处理**: `main.py` 中配置了统一的异常处理器
- **API 错误标准化**: 使用 `success_response` / `error_response` 统一返回格式
- **降级策略**: 多处实现了 fallback 机制 (如 Sina 失败转腾讯)

#### ⚠️ 问题
| 严重程度 | 问题描述 | 位置 |
|----------|----------|------|
| 🔴 高 | 大量裸 `except Exception` 捕获，可能隐藏真正错误 | `sentiment_engine.py`, `data_fetcher.py` |
| 🔴 高 | 线程异常无法被主线程捕获，可能导致静默失败 | `scheduler.py` 中的后台线程 |
| 🟡 中 | 部分 API 未处理超时异常 | `news_engine.py` 的 akshare 调用 |
| 🟡 中 | 数据库操作异常处理不完整 | `database.py` 部分函数 |
| 🟢 低 | 日志记录级别使用不当 | 多处使用 `warning` 记录非警告信息 |

#### 📋 改进建议
1. **精细化异常捕获**: 区分 `ValueError`, `ConnectionError`, `TimeoutError` 等
2. **线程异常处理**: 使用 `concurrent.futures` 或自定义异常钩子
3. **超时统一**: 所有网络请求统一配置超时时间
4. **日志分级**: 错误用 `error`, 信息用 `info`, 调试用 `debug`

---

### 4. 性能优化 (70/100)

#### ✅ 优点
- **缓存机制**: 宏观数据 10 分钟 TTL，新闻 20 分钟刷新
- **后台刷新**: 使用 APScheduler 异步更新数据，不阻塞 API
- **数据库优化**: 使用 WAL 模式，添加索引
- **前端懒加载**: 全市场个股名称懒加载

#### ⚠️ 问题
| 严重程度 | 问题描述 | 影响 |
|----------|----------|------|
| 🔴 高 | 全市场个股 (5000+) 一次性加载到内存 | 内存占用高，启动慢 |
| 🟡 中 | 频繁的数据库连接创建/关闭 | `database.py` 每次操作都新建连接 |
| 🟡 中 | 新闻抓取串行执行，效率低 | `news_engine.py` 循环串行 |
| 🟡 中 | 前端图表数据全量渲染 | `IndexLineChart.vue` 无虚拟滚动 |
| 🟢 低 | 部分 API 响应无缓存 | `quote_detail` 每次都计算 |

#### 📋 改进建议
1. **连接池**: 使用 `sqlite3` 连接池或 `SQLAlchemy`
2. **分页加载**: 全市场个股改为分页/流式加载
3. **并发抓取**: 新闻抓取使用 `asyncio.gather` 或线程池
4. **数据预计算**: 52 周高低等指标可预计算存储

---

### 5. 安全性 (72/100)

#### ✅ 优点
- **SQL 参数化**: 数据库查询使用参数化，防 SQL 注入
- **CORS 配置**: 限制了允许的源
- **无敏感信息硬编码**: 代理配置从环境变量读取
- **输入验证**: 使用 Pydantic 模型验证 API 输入

#### ⚠️ 问题
| 严重程度 | 问题描述 | 位置 |
|----------|----------|------|
| 🔴 高 | 代理密码可能通过日志泄露 | `HTTP_PROXY` 包含认证信息 |
| 🟡 中 | 无 API 认证/鉴权机制 | 所有路由公开访问 |
| 🟡 中 | 新闻详情接口可能存在 SSRF | `news.py` 的 `requests.get(url)` |
| 🟡 中 | 前端 XSS 风险 | `IndexLineChart.vue` tooltip 使用 `v-html` |
| 🟢 低 | 错误信息可能泄露内部路径 | 异常处理器返回原始错误 |

#### 📋 改进建议
1. **代理配置**: 将代理配置移至配置文件，不在代码中打印
2. **API 认证**: 添加 JWT 或 API Key 认证
3. **URL 校验**: 新闻详情接口校验 URL 域名白名单
4. **XSS 防护**: 前端使用 `DOMPurify` 净化 HTML
